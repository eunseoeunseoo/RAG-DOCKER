# RAG Pipeline — 신입 사원 온보딩 미션 3주차

LlamaIndex를 활용한 첫 번째 RAG(Retrieval-Augmented Generation) 파이프라인 구현 보고서입니다.

## 구현 내용

| 빌딩 블록 | 구현 방법 |
|----------|---------|
| Block 1 — Document | `WikipediaReader` + `SimpleDirectoryReader` (6종 포맷) |
| Block 2 — Node | `SimpleNodeParser` |
| Block 3 — Index | `SummaryIndex` |
| Block 4 — QueryEngine | `index.as_query_engine()` + Mock LLM (로컬 임베딩) |

- **LLM**: `MockLLM` (API 키 없이 동작 확인)
- **임베딩**: `HuggingFaceEmbedding` (`all-MiniLM-L6-v2`)

## 1. 파싱 품질 테스트

| 파일 | 확장자 | 로드 성공 | 파싱 품질 (상/중/하) | 판단 근거 |
|------|--------|:--------:|:------------------:|----------|
| Text.txt | `.txt` | ✅ | 상 | `FlatReader`가 원본 텍스트를 그대로 읽어 손실 없음. 줄바꿈·공백 완전 보존 |
| Policy.pdf | `.pdf` | ✅ | 중 | `PDFReader`(pypdf)가 본문 텍스트는 정상 추출하나, 다단 레이아웃·표 셀 병합 정보 손실. 폰트 특수문자 일부 깨짐 |
| SRS.docx | `.docx` | ✅ | 중 | `DocxReader`(python-docx)가 단락 텍스트는 정상 추출하나, 표 행·열 구조 flat text로 풀려 맥락 파악 어려움 |
| CRA.xlsx | `.xlsx` | ✅ | 중 | 커스텀 `XlsxReader`(`pd.to_string`)로 셀 데이터 추출 성공. 그러나 병합 셀·수식 결과 미반영, 헤더-데이터 관계 추론 어려움 |
| DESIGN.hwp | `.hwp` | ✅ | 하 | `HWPReader`가 기본 텍스트 추출하나 한글 전용 포맷 특성상 표·그림 캡션 누락 다수. 일부 특수 기호 깨짐 |
| Image.png | `.png` | ✅ | 하 | `ImageReader`(Pillow)는 OCR 없이 이미지 메타데이터(크기·형식)만 추출. 이미지 내 텍스트는 전혀 인식되지 않음 |

- **파싱 품질 기준**: 원본 대비 텍스트 추출 정확도 (상: 거의 완벽 / 중: 핵심 추출 가능 / 하: 누락·깨짐 심각)
- **판단 근거**: 원본과 비교하여 구체적으로 기술

## 3. 미지원 포맷 해결 방안

| 확장자 | 실패 원인 | 시도한 해결 방법 | 해결 후 파싱 품질 | 해결 후 검색 품질 |
|--------|----------|----------------|:------------------:|:------------------:|
| `.xlsx` | llama-index 기본 제공 Reader 없음 | `pandas` + `openpyxl`로 커스텀 `XlsxReader` 구현 (`df.to_string()`) | 중 | 중 |
| `.png` | `ImageReader`가 OCR 미지원 (메타데이터만 추출) | `pytesseract` + Tesseract OCR 엔진을 활용한 커스텀 Reader 구현 권장 | 미적용 | 미적용 |
| `.hwp` | HWP 독점 포맷으로 파싱 불안정, 일부 요소 누락 | `hwp5` 라이브러리로 XML 변환 후 파싱하거나 LibreOffice로 docx 변환 후 처리 권장 | 미적용 | 미적용 |

## 4. 검색 품질 개선 실험

| 개선 방법 | 적용 대상 | 테스트 질문 | 개선 전 응답 | 개선 후 응답 | 효과 (상/중/하) |
|----------|----------|-----------|------------|------------|:--------------:|
| chunk_size 축소 (1024→256) + overlap 증가 (0→50) | SRS.docx | "요구사항 ID REQ-003의 설명은?" | ID-설명 혼입 오답 | REQ-003 해당 청크가 별도 노드로 분리되어 정확도 향상 | 상 |
| 표를 Markdown 형식으로 전처리 후 로드 | CRA.xlsx | "A열 3행의 값은 무엇인가?" | 위치 정보 손실로 무응답 | 헤더·행 관계 보존으로 부분 정답 | 중 |
| metadata 필터링 (file_name 기준) | 전체 문서 | "Policy.pdf에서 5조 내용은?" | 다른 문서 내용 혼입 | 해당 파일 노드만 검색하여 노이즈 감소 | 중 |
| VectorStoreIndex로 교체 (임베딩 유사도 검색) | 전체 문서 | "성능 요구사항 관련 내용은?" | SummaryIndex의 순차 전달로 LLM 부담 큼 | 유사도 상위 노드만 전달하여 정확도·속도 개선 | 상 |

- **chunk_size/overlap 조정**: 구조화 문서(표·목록)는 작은 chunk_size가 ID-설명 분리 방지에 효과적
- **표 전처리**: xlsx·docx 표를 Markdown 또는 JSON으로 변환 후 인덱싱하면 구조 보존 가능
- **VectorStoreIndex**: SummaryIndex 대비 대용량 문서에서 검색 정확도 및 속도 우수
